# EcomSphere Enterprise Architecture Guide v3.0

This document explains the "Silicon Valley" Layered Architecture implemented in EcomSphere. This structure is designed for high scalability, security, and ease of maintenance.

---

## 🏗️ 1. Backend: The Layered Service Architecture

Your backend is organized into **Layers**. Each layer has a specific job and only talks to certain other layers. This prevents "Spaghetti Code."

### 📂 `src/config/`
*   **What it is**: The system's connection settings.
*   **Role**: Centralizes database connections (MongoDB), mailer settings (Nodemailer), and authentication strategies (Passport).
*   **How it helps**: If you change your database password or mail provider, you only change it here once.

### 📂 `src/models/`
*   **What it is**: The data definitions (Schemas).
*   **Role**: Defines what a "Product," "User," or "Order" looks like in the database.
*   **How it helps**: Ensures that data is consistent. You can't save a product without a price if the model says it's required.

### 📂 `src/services/` (The Brain 🔥)
*   **What it is**: The core business logic.
*   **Role**: Handles heavy lifting like calculating cart totals, filtering products, and processing registration.
*   **How it helps**: Decouples logic from the web. If you ever wanted to move from Express to another framework, your "Brain" (Services) wouldn't need to change.

### 📂 `src/controllers/` (The Traffic Control)
*   **What it is**: The request handlers.
*   **Role**: They receive the request from the user, pass it to a **Service**, and then send back a response (EJS view or JSON).
*   **How it helps**: Keeps the logic out of the web routes. Controllers are "thin" and only manage the flow.

### 📂 `src/routes/` (The Front Door)
*   **What it is**: The URL mapping.
*   **Role**: Connects endpoints (like `/products`) to specific Controller functions.
*   **How it helps**: The `index.js` file here provides a central map of the entire API.

### 📂 `src/middleware/` (The Security Guards)
*   **What it is**: Pre-request checks.
*   **Role**: Checks if a user is logged in (`isAuthenticated`) or if they have Admin rights before letting them pass.
*   **How it helps**: Protects sensitive pages like the Admin Dashboard.

## 🎨 2. Future Storefront (web-next): Feature-Based Architecture

Your future frontend uses a "Modular Feature" approach instead of a flat component list.

### 📂 `src/features/`
*   **What it is**: Domain-driven modules.
*   **Role**: Groups all logic related to a specific part of the app (e.g., `features/cart` holds the "Add to Cart" button and "Cart" calculations).
*   **How it helps**: Makes it easy to find code. If you want to fix a bug in the Shopping Cart, you look in the `cart` feature folder.

### 📂 `src/components/`
*   **What it is**: Atomic, shared UI tokens.
*   **Role**: Holds truly global items like your `Navigation` bar and `Footer`.

## 🔄 3. How They Work Together (The Flow)

1.  **Request**: A customer clicks "Buy" on the Next.js Storefront.
2.  **Route**: The request hits the Backend `src/routes/v1/cart.routes.js`.
3.  **Middleware**: The `auth.middleware.js` checks if the customer is logged in.
4.  **Controller**: The `cart.controller.js` receives the item ID.
5.  **Service**: The `cart.service.js` calculates the new total and saves it to MongoDB using the `Cart.js` **Model**.
6.  **Response**: The Controller sends a success message back to the Frontend.
