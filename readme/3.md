# Operation 3: Random Profile Photos via Built-in Uploads System

## Overview
Replaced the default static `user-avatar.png` on user profiles by assigning them an automated randomized profile image (`1-5.jpg`) originating from a structured generic directory at registration. Incorporated a comprehensive backend capability enabling registered users to overwrite it with their own custom profile photo uploaded securely to the `public/uploads/` directory.

## Changes Made:
- **Initialize Random Photo Setup**: Scaffolded the `public/img/rprofile/` static directory path explicitly to store and supply fallback images cleanly across new user entries.
- **`models/User.js`**: Enhanced the User model document by defining a native `profilePhoto` path initialized gracefully.
- **`controllers/authController.js`**: Rewrote the logic behind handling new user `POST /auth/register` registrations to autonomously pick a random integer between 1 and 5 and map the respective fallback photo string (e.g. `public/img/rprofile/3.jpg`) onto the respective DB tuple before executing `user.save()`.
- **`routes/productRoutes.js`**: Created an independent `POST /profile/photo` route enforcing `multer` parameters exactly tailored to accept incoming form-data. Constrained file checks properly intercept everything but valid image payloads under 5MB.
- **`views/profile.ejs`**: Converted the standard EJS rendering string logic to depend primarily on the respective `user.profilePhoto`. Overhauled the UI layout by crafting a custom clickable 'Change Photo' `<label>` masquerading a hidden input `type="file"`. Upon click, it activates natively, submitting a `multipart/form-data` payload the instant a user approves their system’s file prompt via vanilla Javascript `onchange="this.form.submit()"`.
- **`controllers/productController.js`**: Designed the actual backend image-swap handler `postProfilePhoto()` intercepting the completed `multer` stream. Developed a cleanup phase prior to database commitments, identifying whether the existing user picture belongs securely to `/img/rprofile/` or is a custom upload. If custom, forcefully unlinks it with `fs.unlink()` wiping stale disk footprints.
